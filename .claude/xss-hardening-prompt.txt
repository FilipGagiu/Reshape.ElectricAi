# Task: Add Stored-XSS hardening middleware to {ApiName}

A peer ContentHub microservice was flagged by a pentester (CVSSv4 9.4) for a Stored XSS finding: free-text body fields on JSON write endpoints round-tripped raw `<script>` / event-handler markup into the persistence layer (relational + document stores) and were later rendered as HTML by frontend consumers. The mitigation that shipped — and that you must replicate here — is a request-pipeline middleware that inspects every `application/json` body, rejects requests containing HTML by default, and sanitizes only on endpoints that explicitly opt in.

Treat this as a security hardening task. Apply your project's standard workflow (plan mode, approval gate, code-rule re-read, verification, memory promotion). Review-agent dispatches MUST include "verify CODE.md compliance against the changed files".

## 3. Create the JSON inspector helper

Path: `Sds.ContentHub.{ApiName}.Presentation/Middleware/Helpers/JsonHtmlInspector.cs`

```csharp
using System.Text.RegularExpressions;
using Ganss.Xss;
using Newtonsoft.Json.Linq;

namespace Sds.ContentHub.{ApiName}.Presentation.Middleware.Helpers;

public static class JsonHtmlInspector
{
    private static readonly Regex HtmlTagPattern = new(
        @"<[a-zA-Z!/]",
        RegexOptions.Compiled | RegexOptions.CultureInvariant);

    public static bool ContainsHtml(JToken token) => token.Type switch
    {
        JTokenType.String => ContainsHtml(token.Value<string>()),
        JTokenType.Object => ((JObject)token).Properties().Any(p => ContainsHtml(p.Value)),
        JTokenType.Array  => ((JArray)token).Any(ContainsHtml),
        _ => false
    };

    public static void SanitizeStringsInPlace(JToken token, HtmlSanitizer sanitizer)
    {
        switch (token.Type)
        {
            case JTokenType.String:
                var current = token.Value<string>() ?? string.Empty;
                ((JValue)token).Value = sanitizer.Sanitize(current);
                break;
            case JTokenType.Object:
                foreach (var prop in ((JObject)token).Properties())
                    SanitizeStringsInPlace(prop.Value, sanitizer);
                break;
            case JTokenType.Array:
                foreach (var item in (JArray)token)
                    SanitizeStringsInPlace(item, sanitizer);
                break;
        }
    }

    private static bool ContainsHtml(string? value)
        => !string.IsNullOrEmpty(value) && HtmlTagPattern.IsMatch(value);
}
```

Regex `<[a-zA-Z!/]` is intentionally conservative: it matches the start of a tag, an end-tag, or an HTML/SGML comment, but ignores `<` followed by whitespace/digits (e.g. `"5 < 10"` is not flagged as HTML).

## 4. Create the middleware

Path: `Sds.ContentHub.{ApiName}.Presentation/Middleware/HtmlContentValidationMiddleware.cs`

```csharp
using System.Text;
using Ganss.Xss;
using Newtonsoft.Json;
using Newtonsoft.Json.Linq;
using Sds.ContentHub.Libraries.Core.Logging;
using Sds.ContentHub.{ApiName}.Presentation.Attributes;
using Sds.ContentHub.{ApiName}.Presentation.Middleware.Helpers;

namespace Sds.ContentHub.{ApiName}.Presentation.Middleware;

public class HtmlContentValidationMiddleware
{
    private const string RejectionResponseBody = "{\"error\":\"HTML content is not allowed in request payload.\"}";

    private readonly RequestDelegate _next;
    private readonly HtmlSanitizer _sanitizer;
    private readonly ILogService _logService;

    public HtmlContentValidationMiddleware(RequestDelegate next, HtmlSanitizer sanitizer, ILogService logService)
    {
        _next = next;
        _sanitizer = sanitizer;
        _logService = logService;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        if (!ShouldInspect(context.Request))
        {
            await _next(context);
            return;
        }

        context.Request.EnableBuffering();

        string body;
        using (var reader = new StreamReader(context.Request.Body, Encoding.UTF8, detectEncodingFromByteOrderMarks: false, leaveOpen: true))
        {
            body = await reader.ReadToEndAsync();
        }
        context.Request.Body.Position = 0;

        if (string.IsNullOrWhiteSpace(body))
        {
            await _next(context);
            return;
        }

        JToken token;
        try
        {
            token = JToken.Parse(body);
        }
        catch (JsonReaderException)
        {
            await _next(context);
            return;
        }

        if (!JsonHtmlInspector.ContainsHtml(token))
        {
            await _next(context);
            return;
        }

        var allowsHtml = context.GetEndpoint()?.Metadata.GetMetadata<AllowSanitizedHtmlAttribute>() is not null;
        var endpointPath = context.Request.Path.Value ?? string.Empty;

        if (!allowsHtml)
        {
            _logService.LogWarning(
                $"Rejected request containing HTML content. Path='{endpointPath}', Method='{context.Request.Method}'.",
                new InvalidOperationException("HTML content detected in request payload"));

            context.Response.StatusCode = StatusCodes.Status400BadRequest;
            context.Response.ContentType = "application/json";
            await context.Response.WriteAsync(RejectionResponseBody);
            return;
        }

        JsonHtmlInspector.SanitizeStringsInPlace(token, _sanitizer);
        var sanitizedBytes = Encoding.UTF8.GetBytes(token.ToString(Formatting.None));
        context.Request.Body = new MemoryStream(sanitizedBytes);
        context.Request.ContentLength = sanitizedBytes.Length;

        _logService.LogInformation(
            $"Sanitized HTML content in request payload. Path='{endpointPath}', Method='{context.Request.Method}'.");

        await _next(context);
    }

    private static bool ShouldInspect(HttpRequest request)
    {
        if (HttpMethods.IsGet(request.Method)
            || HttpMethods.IsDelete(request.Method)
            || HttpMethods.IsHead(request.Method)
            || HttpMethods.IsOptions(request.Method))
        {
            return false;
        }

        if (request.ContentLength is null or 0)
        {
            return false;
        }

        var contentType = request.ContentType;
        return !string.IsNullOrEmpty(contentType)
               && contentType.StartsWith("application/json", StringComparison.OrdinalIgnoreCase);
    }
}
```

Notes if the target service differs from the reference implementation:

- The logger abstraction shown is `Sds.ContentHub.Libraries.Core.Logging.ILogService`. If your service uses a different logger (`ILogger<T>`, a custom wrapper, etc.), swap the type and keep both log lines: a `Warning` on rejection (with the rejection reason as a faux exception per the ContentHub convention) and an `Information` on sanitization.
- If your service has no JSON-body endpoints (pure gRPC + multipart), this middleware still belongs in the pipeline as defense-in-depth: `ShouldInspect` will skip every request and the runtime cost is negligible.
- Body buffering via `EnableBuffering` is required because downstream controllers / model binders must be able to re-read the stream. Do not remove it.

## 5. Register in Startup

In `Sds.ContentHub.{ApiName}.Presentation/Startup.cs`:

**In `ConfigureServices` (singleton registration):**

```csharp
services.AddSingleton<HtmlSanitizer>(_ => new HtmlSanitizer());
```

Use the Ganss.Xss default whitelist. Do not pass a custom config unless a specific endpoint demonstrably needs an expanded tag/attribute set.

**In `Configure` (pipeline placement — order is load-bearing):**

```csharp
app.UseRouting();
app.UseAuthentication();
app.UseAuthorization();
app.UseMiddleware<HtmlContentValidationMiddleware>();   // <-- here
// ... any custom context-building middleware (e.g. tenant resolver, request enricher) ...
// ... controller / GraphQL / SignalR endpoint mapping ...
```

Placement rules:

- AFTER `UseRouting()` — required so `context.GetEndpoint()` resolves and the `[AllowSanitizedHtml]` attribute is visible via endpoint metadata.
- AFTER `UseAuthorization()` — so unauthenticated/unauthorized requests are rejected before payload inspection runs (no point sanitizing payloads that won't be processed).
- BEFORE any tenant- or context-building middleware — so a poisoned payload never reaches scope-resolution logic that may log or echo it.
- BEFORE controllers / GraphQL / SignalR endpoint mapping.

If the service has its own `ErrorLoggingMiddleware` or equivalent global exception handler, that should sit earlier in the pipeline (before `UseRouting`). The XSS middleware should not be wrapped by retry / circuit-breaker decorators.

## 6. Tests

Create two test files in the target service's unit-test project (path convention typically `Sds.ContentHub.{ApiName}.Core.UnitTests/Presentation/Middleware/`).

### `JsonHtmlInspectorTests.cs` — pure unit tests

Required scenarios:

- `ContainsHtml` returns true for: `<script>alert(1)</script>`, `<img src=x onerror=alert(document.domain)>`, `hello </a> world`, `<!-- comment -->`, `plain text <b>bold</b> tail`.
- `ContainsHtml` returns false for: `"5 < 10"`, `"&lt;b&gt;bold&lt;/b&gt;"`, empty string, `"a < b > c"`, `"trailing <"`.
- `ContainsHtml` on a null `JValue` returns false.
- `ContainsHtml` on numbers, bools, null returns false.
- `ContainsHtml` recurses through nested object + array and finds HTML deep inside.
- `ContainsHtml` on nested clean payload returns false.
- `SanitizeStringsInPlace` strips `<script>` and its contents.
- `SanitizeStringsInPlace` preserves whitelisted tags like `<b>hi</b>`.
- `SanitizeStringsInPlace` strips event handlers like `onerror=alert(1)`.
- `SanitizeStringsInPlace` recurses into nested objects/arrays.
- `SanitizeStringsInPlace` leaves numbers, bools, and nulls untouched.

### `HtmlContentValidationMiddlewareTests.cs` — `Microsoft.AspNetCore.TestHost` integration tests

Spin up an in-process `TestServer` with two endpoints: one bare, one decorated via `.WithMetadata(new AllowSanitizedHtmlAttribute())`. Both echo the inbound body back as the response so assertions can inspect the post-middleware payload.

Required scenarios:

- `text/plain` content-type body containing HTML → passes through untouched (middleware ignores non-JSON).
- Unparseable JSON body → passes through untouched (middleware does not crash).
- Clean JSON on the bare endpoint → 200, body echoed unchanged.
- HTML in JSON on the bare endpoint → 400, response body matches the rejection envelope (`HTML content is not allowed`), and crucially the response MUST NOT echo the attack payload back.
- Clean JSON on the `[AllowSanitizedHtml]` endpoint → 200, body echoed unchanged.
- HTML in JSON on the `[AllowSanitizedHtml]` endpoint → 200, dangerous tags stripped, safe tags preserved (`<b>hi</b>` survives, `<script>bad</script>` is gone).
- Event handlers (`onerror=alert(...)`) on the sanitize endpoint → stripped from the persisted payload.
- Nested HTML deep in object/array on the bare endpoint → 400.

Mock `ILogService` (or your service's logger abstraction) — no log assertions required; the middleware logs are operational, not behavioural.

## 7. Opt endpoints into sanitization (don't sprinkle blindly)

Default = reject. Only decorate an action or controller with `[AllowSanitizedHtml]` when the endpoint genuinely needs to accept formatted text (rich descriptions, marketing copy, user-authored notes).

Inventory every `POST` / `PUT` / `PATCH` controller action in the service and decide per-endpoint:

- Plain-text fields (names, codes, IDs, enums, dates, numbers) → no decoration; rejection is correct.
- Free-text rich content (descriptions, notes, long-form text fields) → `[AllowSanitizedHtml]` on the action (preferred — narrowest scope) or controller (only if every action on it accepts HTML).
- Mixed endpoints (some fields plain, some rich) → still decorate; sanitization is a no-op on fields that never carry HTML, and rejection on the rich field would be a false positive.

List every endpoint you decorate in the PR description with a one-sentence rationale (which field needs HTML, why).

## 8. Verification checklist before claiming done

- `dotnet build` clean — only the project's pre-existing baseline warnings should appear. Cross-check against the project's documented expected-warnings list before claiming a regression.
- All new tests pass:
  `dotnet test --filter "FullyQualifiedName~HtmlContentValidationMiddlewareTests|JsonHtmlInspectorTests"`
- Manual smoke against a dev instance:
  - `POST` JSON body `{"foo":"<script>alert(1)</script>"}` to a non-decorated endpoint → expect `400` with the rejection envelope; the response MUST NOT echo the attack string back.
  - Same payload to a decorated endpoint → expect `200`, and the downstream stored value MUST NOT contain `<script>` or `onerror`.
  - `GET` with HTML in the query string → middleware skips (any rejection MUST come from controller-level validation if needed; query-string XSS is out of scope here).
- Confirm gRPC / SignalR / multipart endpoints are untouched (the content-type filter excludes them).
- Confirm GraphQL endpoints (`POST /…/graphql`, `application/json`) ARE inspected. This is intended: GraphQL mutation args can carry XSS via string variables, and rejection is the safe default. If a GraphQL mutation legitimately needs to accept HTML, either decorate the GraphQL endpoint registration with `[AllowSanitizedHtml]` or split the GraphQL route onto its own pipeline branch with the attribute applied.

## 9. Out of scope (explicitly)

Document each of the following in the PR description so reviewers do not expect coverage:

- Query-string XSS — no body to inspect; controller / GraphQL validators handle their own argument sanitization.
- Stored XSS reading data BACK out — this middleware is ingress-only. Egress safety is the frontend's CSP plus output-encoding responsibility.
- File uploads / multipart bodies — skipped by content-type filter; covered by separate antivirus + extension allowlist.
- WebSocket frames after the upgrade handshake — protocol-level, not HTTP-body-level.
- gRPC payloads — binary protobuf, distinct attack surface, handled by gRPC-side validation.
